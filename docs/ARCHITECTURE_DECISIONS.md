# Architecture Decisions

## Express/TypeScript v1

The v1 scaffold stays on Express + TypeScript + Postgres. Do not migrate to FastAPI during v1 workflow work.

## Communication-First Product Shape

The shared inbox, contact summary, quote workflow, calls, tasks, and activity timeline are one unified staff workspace, not separate tools.

## Unified Timeline Rule

Messages, calls, notes, tasks, quote events, and service site updates write to `activities` for the contact.

## Provider Abstraction

CRM logic is provider-neutral. SMS/call provider payloads normalize through adapters in `message_provider` and `sms_provider_registry`, then flow through `integration_service`.

## Provider IDs

Internal CRM IDs are UUIDs generated by this app. External IDs stay in provider fields such as `provider_message_id`, `provider_call_id`, `provider_conversation_id`, `provider_name`, or `external_references`.

## Support Tables

`message_templates`, `phone_routing_settings`, `quote_defaults`, `integration_settings`, and `external_references` are v1 support tables and should remain unless a specific migration replaces them.

## Auth

Auth is scaffold-only. Admin/provider screens are demo surfaces until real sessions, SSO, or password auth are implemented.

## Scope Control

Testing passes should not add features. Feature work should use small diffs and avoid unrelated refactors.
