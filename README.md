# Communication-First CRM

Standalone web CRM workspace for contacts, service sites, shared SMS inbox, call logging, quotes, tasks, admin settings, and one unified contact activity timeline.

This project is intentionally separate from earlier phone integration starters. The sample quote/customer data is generic demo data so the workflow has realistic context without client-specific branding.

## Current Contract

V1 is intentionally built with `Express` + `TypeScript`, backed by Postgres and REST APIs. That is the active scaffold for proving the communication-first workflow quickly and keeping the product stable while the workflow is being refined.

The original FastAPI direction is not a blocker for this release. A FastAPI migration can be considered later if it creates clear operational value, but it is optional and not required for finishing the demo-ready v1.

## Scope Notes

Core CRM tables remain the primary domain model:

- `contacts`
- `accounts`
- `service_sites`
- `conversations`
- `messages`
- `calls`
- `quotes`
- `quote_versions`
- `quote_line_items`
- `activities`
- `tasks`
- `attachments`
- `tags`
- `contact_tags`
- `users`

This v1 also includes support tables that are intentionally kept because they are already wired into admin and workflow surfaces:

- `message_templates`
- `phone_routing_settings`
- `quote_defaults`
- `integration_settings`
- `external_references`

Treat these as v1 support tables, not as a replacement for the core CRM model.

## Auth Status

Authentication is scaffold-only in this version. The current `/auth/session` and `/auth/login` flows are for local/demo use and are not production-secure.

Before production, replace scaffold auth with one of:

- signed server sessions
- SSO
- password-based auth with hashing, session management, authorization checks, and logout

Admin and provider settings should be treated as demo configuration surfaces until that work is complete.

## Local Setup

```bash
cp .env.example .env
npm install
npm run db:setup
npm run dev:run
```

Open `http://127.0.0.1:3000/`.

Load sample data:

```bash
npm run db:seed
```

Run a lightweight workflow smoke test while the app is running:

```bash
npm run test:workflow
```

## Deploy

For remote testing from another device, deploy this standalone repo to a host such as Railway, Render, or Fly.io and attach Postgres.

Build command:

```bash
npm install && npm run build
```

Start command:

```bash
npm start
```

Required environment variable:

```bash
DATABASE_URL=postgres://...
```

For a temporary staging demo where the Gecko needs sample data:

```bash
ALLOW_DEMO_SEED=true
```

After the database is available, run:

```bash
npm run db:setup
```

## Provider Hooks

Live telephony/SMS is intentionally abstracted. Add provider implementations in:

- `src/services/integration_service.ts`
- `src/services/notification_service.ts`
- `src/services/call_service.ts`
- `src/services/message_service.ts`

Provider IDs stay in `provider_message_id` and `provider_call_id`; internal IDs remain UUIDs generated by this app.

## Future External Linking

Use `external_references` to connect standalone CRM records to external systems without merging databases too early.

Example:

```json
{
  "internalType": "quote",
  "internalId": "crm-quote-uuid",
  "externalSystem": "legacy-system",
  "externalType": "estimate",
  "externalId": "EXT-DEMO-EST-0001",
  "metadata": {
    "source": "legacy-estimator"
  }
}
```
